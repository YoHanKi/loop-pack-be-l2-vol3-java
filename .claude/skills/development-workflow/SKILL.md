# 개발 워크플로우

---

## 증강 코딩 원칙

**방향성 및 주요 의사 결정은 개발자에게 제안만 할 수 있으며, 최종 승인된 사항을 기반으로 작업을 수행.**

1. **중간 결과 보고**: 단계별 진행 상황 명확히 보고. 예상치 못한 문제 발생 시 즉시 보고
2. **설계 주도권 유지**: 아키텍처 변경, 새로운 패턴 도입 시 사전 협의 필수
3. **명확한 커뮤니케이션**: 불확실한 부분은 가정하지 않고 질문

---

## TDD 워크플로우 (Red → Green → Refactor)

### Red — 실패하는 테스트 먼저 작성

1. 요구사항 분석 → 구현할 기능 명세 파악
2. `// given / when / then` 패턴으로 테스트 작성
3. 테스트 실행 → **실패 확인**

체크리스트:
- [ ] 테스트 케이스가 요구사항을 정확히 반영하는가?
- [ ] `@DisplayName` 이 명확한가?
- [ ] 테스트가 실패하는가? (Red 확인)

테스트 작성 패턴 → `/testing` 스킬 참조

---

### Green — 테스트를 통과하는 최소 코드 작성

1. 테스트를 통과하는 **최소한의 코드** 구현 (오버엔지니어링 금지)
2. 테스트 실행 → **통과 확인**
3. 엣지 케이스 테스트 추가 및 구현

체크리스트:
- [ ] 모든 테스트가 통과하는가? (Green 확인)
- [ ] 불필요한 코드가 없는가?

---

### Refactor — 코드 품질 개선

1. 코드 리뷰 → 개선 가능한 부분 파악
2. 리팩토링 (아래 체크리스트 기준)
3. 테스트 실행 → **모든 테스트 통과 확인**
4. 반복

리팩토링 대상:
- 과도하게 분리된 private 메서드 제거
- Value Object로 책임 이동 (검증 로직 캡슐화)
- Unused import 제거
- N+1 쿼리, 불필요한 DB 조회 제거

체크리스트:
- [ ] 불필요한 private 메서드를 제거했는가?
- [ ] Unused import를 제거했는가?
- [ ] 모든 테스트가 통과하는가?

---

## 핵심 규칙

```
❌ 실제 동작하지 않는 Mock 데이터로만 테스트 통과
❌ null 반환  →  Optional 또는 orElseThrow
❌ System.out.println  →  @Slf4j log.info/warn/error
❌ @Disabled 또는 assertion 약화 (isNotNull 만으로 종료)
```

---

## 우선순위

1. **실제 동작하는 해결책** — Mock 데이터가 아닌 실제 데이터로 검증
2. **null-safety, thread-safety** — Optional, 불변 객체, 동시성 고려
3. **테스트 가능한 구조** — 의존성 주입, 인터페이스 기반
4. **기존 패턴 일관성** — 코드베이스 기존 패턴 준수

## 코드 리뷰 체크리스트

### 기능
- [ ] 요구사항을 모두 만족하는가?
- [ ] 엣지 케이스를 처리하는가?
- [ ] 에러 처리가 적절한가?

### 테스트
- [ ] 단위/통합/E2E 테스트가 작성되었는가?
- [ ] 모든 테스트가 통과하는가?

### 아키텍처
- [ ] 레이어 의존성 방향이 올바른가?
- [ ] App/Facade/Service 규칙을 준수하는가?
- [ ] VO, 네이밍 컨벤션을 준수하는가?

### 성능
- [ ] N+1 쿼리 문제가 없는가?
- [ ] 불필요한 DB 조회가 없는가?

### 보안
- [ ] 입력 검증이 적절한가?
- [ ] 민감 정보가 로그에 남지 않는가?
