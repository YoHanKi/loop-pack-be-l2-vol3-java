외부 시스템 연동 설계를 분석할 때 반드시 다음 흐름을 따른다.

---

### 0️⃣ 분석 시작 전: 컨텍스트 수집 (질의응답)

분석에 들어가기 전 아래 항목이 명확하지 않으면 개발자에게 질문한다.
추측으로 빈칸을 채우지 않는다.

확인할 항목:
- 외부 시스템의 통신 방식은 무엇인가? (동기 응답 / 비동기 콜백 / 폴링)
- 외부 시스템의 응답 지연 범위와 실패율을 알고 있는가?
- 외부 시스템 호출 결과가 내부 상태 변경과 원자적으로 묶여야 하는가?
- 동일 요청이 두 번 실행됐을 때 외부 시스템은 어떻게 동작하는가?
- 외부 시스템 장애 시 내부 처리를 중단할 것인가, 아니면 계속 진행할 것인가?

---

### 1️⃣ 기능이 아니라 "불확실성" 관점으로 재해석한다

단순 호출 순서를 요약하지 않는다.
외부 시스템은 항상 다음을 만족한다고 가정한다:

- 지연될 수 있다
- 실패할 수 있다
- 중복 실행될 수 있다
- 성공했지만 응답이 유실될 수 있다
- **부분 성공할 수 있다** — 일부만 처리되고 나머지는 누락되는 경우
- **상태가 내부와 달라질 수 있다** — PG가 처리 완료했는데 우리 DB는 PENDING

현재 설계가 이러한 불확실성을 어떻게 다루는지 설명한다.

---

### 2️⃣ 트랜잭션 경계를 검증한다

외부 호출이 트랜잭션 내부에 존재하는지 확인한다.

다음 질문을 반드시 포함한다:
- 외부 호출 실패 시 내부 상태는 어떻게 되는가?
- 내부 커밋 이후 외부 호출 실패 시 복구 가능한가?
- 외부 성공 후 내부 실패 시 상태는 어떻게 정합성을 유지하는가?

**추가 확인 — DB 커넥션 점유 문제:**

외부 호출이 트랜잭션 내부에 있으면 외부 시스템의 지연(1~5초)이 DB 커넥션 점유로 이어진다.
커넥션 풀(HikariCP 기본 40개)이 고갈되면 전체 서비스가 응답 불가 상태가 된다.
다음을 점검한다:

- 외부 호출 이전에 트랜잭션을 커밋할 수 있는가?
- Payment(PENDING) 저장 → 커밋 → PG 호출 → 결과에 따라 별도 트랜잭션으로 업데이트 구조인가?
- `@Transactional(propagation = REQUIRES_NEW)` 활용 시 자기 호출(self-invocation)로 인해 AOP가 동작하지 않는 상황은 없는가?

**출력 예시:**
```
트랜잭션 경계 분석:
PaymentFacade.requestPayment()
  ├─ [TX 1] PaymentApp.createPendingPayment() → 커밋
  ├─ [TX 없음] PgPaymentGateway.requestPayment() ← 외부 호출
  └─ [TX 2] PaymentApp.updateRequested() → 커밋

위험 지점:
- TX 1 커밋 후 PG 호출 실패 → Payment(PENDING) 고착 가능
- PG 호출 중 JVM crash → TX 2 실행 불가 → 정합성 불일치
```

---

### 3️⃣ 상태 기반으로 구조를 다시 본다

호출 흐름이 아니라 상태 전이를 중심으로 설명한다.
내부 도메인 상태와 외부 시스템 상태를 분리해서 정리한다.
두 상태가 어긋날 수 있는 지점을 명시한다.

추가 확인:
- 외부 시스템의 상태를 폴링(조회 API)으로 동기화할 수 있는가?
- 상태 전이 중 되돌릴 수 없는(irreversible) 전이가 있는가?
  - 있다면 해당 전이 이후의 보상 방법이 설계되어 있는가?
- 내부 상태와 외부 상태가 불일치한 채로 얼마나 허용 가능한가? (Eventually Consistent 허용 범위)

**출력 예시:**
```
내부 상태 (Payment):  PENDING → REQUESTED → COMPLETED
                                           → FAILED
외부 상태 (PG):       PROCESSING → COMPLETED
                                 → CREDIT_LIMIT_EXCEEDED

불일치 가능 지점:
- Payment=REQUESTED, PG=COMPLETED (콜백 지연) → 일시적 불일치
- Payment=PENDING, PG=COMPLETED (Timeout) → 수동 동기화 없으면 영구 불일치
```

---

### 4️⃣ 중복 요청 및 재시도 가능성을 분석한다

네트워크 재시도 상황을 가정한다.
동일 요청이 두 번 실행될 경우의 문제를 설명한다.

**멱등성 키(Idempotency Key) 설계 확인:**

단순히 "멱등성을 고려하는가?"가 아니라 구체적으로 점검한다:
- 외부 호출 시 Idempotency-Key 헤더를 전달하는가?
- 외부 시스템이 동일 키를 받았을 때 이전 결과를 그대로 반환하는가?
- 우리 내부에서 동일 paymentId로 PG를 두 번 호출했을 때 어떻게 되는가?
- Retry 설정이 있다면 재시도된 요청과 최초 요청이 모두 처리될 위험은?

**Retry 적용 시 추가 확인:**
- 결제와 같이 side-effect가 있는 호출에 Retry를 적용하는가?
- Retry 대상 예외가 명확히 정의되어 있는가? (일시적 실패만 Retry해야 함)
- Exponential Backoff 설정이 있는가? (무작정 재시도는 DoS와 같음)

---

### 5️⃣ 장애 시나리오를 최소 5가지 이상 생성한다

정상 흐름보다 실패 흐름을 우선한다.

**기본 시나리오 (반드시 포함):**
1. **외부 시스템 응답 지연 (Timeout)** — 우리 시스템은 어떻게 반응하는가? 스레드가 블로킹되는가?
2. **외부 시스템 완전 다운** — Circuit Breaker 없이 계속 호출하면 어떻게 되는가?
3. **성공 응답 후 콜백/후처리 실패** — 외부는 성공, 내부는 미반영. 어떻게 복구하는가?
4. **중복 요청 처리** — 동일 결제 요청이 두 번 실행됐을 때 이중 결제가 발생하는가?
5. **부분 성공** — 외부 시스템이 일부만 처리하고 나머지는 유실된 경우
6. **비정상 응답자** — 외부를 가장한 악의적 호출 (위조 콜백, 금액 조작 등)

각 장애 상황에서:
- 데이터 정합성 — 내부 상태가 일관된 상태인가?
- 복구 가능성 — 운영팀이 개입 없이 자동 복구되는가?
- 사용자 영향 — 사용자는 어떤 경험을 하는가?

---

### 6️⃣ 관찰 가능성(Observability)을 점검한다

외부 연동 설계에서 자주 빠지는 요소다.
장애는 반드시 관찰 가능해야 빠르게 대응할 수 있다.

확인 항목:
- **로깅**: 외부 호출 요청/응답/오류가 로그로 남는가? 민감 정보(카드번호)가 마스킹되는가?
- **메트릭**: Circuit Breaker 상태, 호출 성공/실패율, 응답 시간 p99가 Grafana에서 확인 가능한가?
  - `resilience4j_circuitbreaker_state` — CB 상태 추이
  - `resilience4j_circuitbreaker_failure_rate` — 실패율 추이
  - `http_server_requests_seconds` p99 — 외부 연동 영향 여부
- **알람**: 실패율 N% 초과, CB Open 전환, 콜백 미수신 N분 경과 시 알람이 설정되어 있는가?
- **추적**: 요청 ID(traceId)가 외부 시스템 호출 로그와 연계되어 있는가?

---

### 7️⃣ 해결책은 정답처럼 제시하지 않는다

현재 구조의 장점과 리스크를 분리한다.
대안 구조가 있다면 선택지 형태로 제시한다.

**선택지 제시 형식:**
```
[현재 구조의 리스크]
- 외부 호출이 트랜잭션 내부에 있어 DB 커넥션 고갈 가능

[개선 선택지]
A. 동기 호출 유지 + 트랜잭션 분리
   - Payment 저장 커밋 → PG 호출 → 결과 반영
   - 복잡도 낮음, 일시적 불일치 허용

B. 비동기 이벤트 전환 (Kafka)
   - 결제 요청을 이벤트로 발행, 비동기 처리
   - 운영 복잡도 증가, 추적 어려움

C. Saga 패턴 (보상 트랜잭션)
   - 각 단계가 실패하면 이전 단계를 되돌리는 보상 로직
   - 분산 트랜잭션 문제 해결, 구현 복잡도 높음

각 선택지의 복잡도와 운영 부담을 함께 설명한다.
```

### 8️⃣ 톤 & 스타일 가이드

- 코드 레벨 수정안을 직접 제시하지 않는다.
- 설계를 비판하지 말고 리스크를 드러내는 리뷰 톤을 유지한다.
- 외부 시스템은 항상 신뢰할 수 없다는 전제를 유지한다.
- 구현보다 책임, 경계, 상태 일관성을 중심으로 분석한다.
- 명확하지 않은 부분은 "확인 필요"로 표시하고 개발자에게 질문한다.
- 각 섹션 마지막에 "결정 필요 사항"을 명시해서 개발자가 선택할 수 있게 한다.
